## Goal
Make all content views share one constant width, controlled by a single CSS variable (not hardcoded in a dozen places), while keeping the existing responsive behavior for mobile/tablet and the centered max-width behavior for large monitors.

## Root Cause
The value `1200px` is duplicated across **7 locations** with 5 different class names (`.container`, `.landing-container`, `.kasse-container`, `.lz-detail-main`, `.docs-content`, plus an inline `main` selector). They all happen to use 1200px today, so most views *look* consistent — **except** `device-pairings.html`, which overrides to `1000px` via `.pairing-page`, causing a visible width jump when navigating Devices → Reader → Pairing → NFC-Tags. There is no single source of truth, so any future edit to one class drifts from the others.

## Approach: CSS variable + dedup (minimal, low-risk)

### 1. Add a single source-of-truth variable
**`static/css/style.css`** — in `:root` (line 3):
```css
--content-max-width: 1200px;
```
This is the only place the number lives. Change it once → every view updates.

### 2. Point every container class at the variable (7 edits)
| File | Class | Current | New |
|---|---|---|---|
| `static/css/style.css:51` | `.container` | `max-width: 1200px` | `max-width: var(--content-max-width)` |
| `static/css/landing.css:12` | `.landing-container` | `max-width: 1200px` | `max-width: var(--content-max-width)` |
| `static/css/kasse.css:12` | `.kasse-container` | `max-width: 1200px` | `max-width: var(--content-max-width)` |
| `static/css/laufzettel-detail.css:115` | `.lz-detail-main` | `max-width: 1200px` | `max-width: var(--content-max-width)` |
| `static/css/docs.css:301` | `.docs-content` | `max-width: 1200px` | `max-width: var(--content-max-width)` |
| `templates/guest-laufzettel-detail.html:40` | inline `main` | `max-width: 1200px` | `max-width: var(--content-max-width)` |

### 3. Fix the one visible inconsistency
**`templates/device-pairings.html`** — the `<div class="container pairing-page">` (line 250) gets narrowed to 1000px by the inline `.pairing-page { max-width: 1000px }` rule (line 10-12). Remove that `max-width` declaration from `.pairing-page` (keep the class for any other styling it carries). Now Pairing matches Reader and NFC-Tags at the standard width.

### 4. Leave responsive behavior untouched
The existing media queries already handle scaling down:
- `style.css:446` — at `≤768px`, `.container` becomes `max-width: 100vw; padding: 12px`
- `style.css:571` — at `≤480px`, padding drops to `10px`
- `landing.css`, `kasse.css`, etc. have their own `≤768px` / `≤480px` overrides

These override `max-width` directly and will continue to win the cascade over the variable. No breakpoint changes needed.

## Explicitly out of scope (intentionally different, kept as-is)
- **Narrow form/card pages** — `register.html` (520px), `guest-laufzettel-form.html` (500px), `bug-report.html` (540px), `offline.html`. These are single-purpose centered cards (login/registration), not content views. Stretching them to 1200px would look wrong.
- **Docs sidebar grid** — `docs-layout.html` uses a `290px sidebar + 1fr` grid (separate app on port 8001). Only its `.docs-content` max-width is pointed at the variable; the grid layout itself is untouched.
- **Component-level widths** — inline `max-width` on individual forms/modals/inputs (e.g. `buchhaltung.html` form at 480px, various modals at 400-560px). These are component sizing, not page-width inconsistencies.

## Result
Every content view (admin dashboard, devices, pairing, laufzettel, katalog, buchhaltung, mitglieder, member pages, landing, kasse, guest detail, public laufzettel) renders at exactly `var(--content-max-width)`, centered, with the same padding model it had before. Navigating between views produces zero width shift. To change the width globally (e.g. 1400px for bigger monitors), edit one line in `:root`.
